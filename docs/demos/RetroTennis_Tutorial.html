Retro Tennis

Ein architektonisches Demo-Projekt für den Game Builder MVC

Einführung

Willkommen beim Retro Tennis Demo! Dieses Projekt ist (aus Gründen des Markenschutzes 😉) eine liebevolle Hommage an die frühesten Videospiele der Geschichte.

Anhand dieses Projektes kannst du live im Game-Builder sehen, wie Sprites, kontinuierliche Game-Loops (`onFrame`) und physikalische Grundgesetze (Abpraller/Bounces) mithilfe des visuellen Flow-Editors perfekt ineinandergreifen.

Blueprint Stage

Globale Werkzeuge & Manager (Blueprint)

Wir nutzen die Blueprint-Stage, um unsere gesamte grundlegende Architektur global sichtbar abzulegen. Wir nutzen echte technische System-Klassen (Components), um den Flow zu lenken:

  • GameLoop (Klasse: TGameLoop): Konfiguriert den Loop (targetFPS) und triggert jeden Frame den MainGameLoop Task.
  • InputController (Klasse: TInputController): Bündelt alle nativen Tastatur-Eingaben (onKeyDown ruft HandleInput auf).
  • GameState (Klasse: TGameState): Die zentrale Zustandsmaschine als Komponente (Status: 'menu' | 'playing'). Alle Loops prüfen ihren Wert.
Main Stage

Der 100% visuelle Spielfluss (Keine versteckte Logik!)

Auf dem Spielfeld (Main Stage) liegen die Darsteller (Paddles, Ball) und ein großer grüner StartButton (TButton).

Anstatt das Spiel "unsichtbar" über ein System-Event zu starten, ist der Start eine logische, vom Nutzer visuell nachvollziehbare Kette im Flow-Editor:

StartButton.onClick ➔ StartGameTask
⬇
Action: SetStatePlaying (Setzt GameState.state = 'playing')
Action: StartBallX (Gibt dem Ball velocityX)
Action: StartBallY (Gibt dem Ball velocityY)
Action: HideStartButton (Macht den Button unsichtbar)

Sobald GameState.state == 'playing' ist, beginnt der globale GameLoop (aus dem Blueprint) zu greifen, weil seine Condition nun erbracht ist. Der Loop ruft jeden Frame PlayStateLoop > CheckWallCollisions auf.

Dort greifen dann die Abpraller-Actions (BounceX und BounceY), welche simpel die Geschwindigkeit invertieren (type: "negate").

Der Pascal-Code Beweis

Im Flow Editor generiert der Pascal Code Generator aus dieser sauberen Top-Down-Architektur Folgendes: 

PROGRAM RetroTennis;

VAR { STAGE COMPONENTS }
  GameManager: TComponent;
  InputController: TComponent;
  Ball: TSprite;
  GameState: Integer; { 0=Menu, 1=Play, 2=Gameover }

PROCEDURE MainGameLoop; 
BEGIN 
  IF GameState = 1 THEN BEGIN
    PlayStateLoop (); 
  END; 
END; 

PROCEDURE HandleInput; 
BEGIN 
  IF GameState = 1 THEN BEGIN
    ProcessPaddleMovement (); 
  END; 
END; 

{ OBJECT EVENT HANDLERS }
PROCEDURE GameManager.onFrame();
BEGIN
  MainGameLoop();
END;

PROCEDURE InputController.onKeyDown();
BEGIN
  HandleInput();
END;

Daran erkennst du: Wahre Spiele-Entwicklung basiert auf sauberen Zuständen und Managern, die Aufgaben delegieren.
